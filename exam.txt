ТЕХНИЧЕСКОЕ ЗАДАНИЕ (ТЗ)
на разработку клиент-серверного веб-приложения "Mindful Meals"

1. ОБЩИЕ СВЕДЕНИЯ

1.1. Название проекта: Mindful Meals (Планировщик питания)
1.2. Цель проекта: Разработка интерактивного веб-приложения для персонализированного планирования здорового питания, отслеживания калорий и генерации списков покупок.
1.3. Тип приложения: Клиент-серверное веб-приложение (Single Page Application - SPA).
1.4. Срок реализации: (Указывается Заказчиком/Исполнителем)
1.5. Целевая аудитория: Люди, ведущие здоровый образ жизни, следящие за питанием, а также пользователи с диетическими ограничениями.

2. ТРЕБОВАНИЯ К ФУНКЦИОНАЛУ

2.1. Клиентская часть (Frontend)
    2.1.1. Регистрация и Авторизация:
        * Регистрация по email/паролю.
        * Авторизация и восстановление пароля.
        * Возможность входа через социальные сети (опционально).
    2.1.2. Настройка Профиля:
        * Ввод личных данных: пол, возраст, рост, вес.
        * Установка цели: похудение, поддержание, набор массы.
        * Выбор диетических предпочтений/ограничений (веган, без глютена, аллергии и т.д.).
        * Расчет базовой нормы калорий и БЖУ.
    2.1.3. Планировщик Питания (Главный экран):
        * Отображение недельного календаря.
        * Возможность добавления/удаления/редактирования приемов пищи (завтрак, обед, ужин, перекусы).
        * Поиск и добавление блюд/продуктов из базы данных.
        * Отображение суммарного потребления калорий и БЖУ за день.
        * Функция "Сгенерировать план" на основе цели и предпочтений пользователя.
    2.1.4. База Рецептов/Продуктов:
        * Поиск по названию, ингредиентам, категории.
        * Возможность добавления собственных рецептов пользователем.
        * Просмотр подробной информации о блюде: калорийность, БЖУ, список ингредиентов.
    2.1.5. Список Покупок:
        * Автоматическая генерация списка покупок на основе выбранного недельного плана.
        * Возможность ручного редактирования списка.
        * Разделение списка по категориям (молочные продукты, овощи, мясо и т.д.).

2.2. Серверная часть (Backend)
    2.2.1. API:
        * Разработка RESTful API для взаимодействия с клиентской частью.
        * Эндпоинты для управления пользователями (регистрация, авторизация, профиль).
        * Эндпоинты для управления планами питания и записями о приемах пищи.
        * Эндпоинты для работы с базой рецептов/продуктов.
    2.2.2. База Данных:
        * Хранение данных пользователей (профили, цели).
        * Хранение данных о рецептах и продуктах с их нутриентным составом.
        * Хранение недельных планов питания и записей о потребленных продуктах.
    2.2.3. Бизнес-логика:
        * Алгоритм расчета суточной нормы калорий и БЖУ (например, формула Миффлина-Сан-Жеора).
        * Логика генерации плана питания, учитывающая ограничения и цели.
        * Логика формирования списка покупок.

3. ТЕХНОЛОГИЧЕСКИЕ ТРЕБОВАНИЯ

3.1. Стек технологий (Предложение):
    * Frontend: React/Vue/Angular (например, **React** с TypeScript).
    * Backend: Node.js (Express) / Python (Django/Flask) / Go (например, **Node.js с Express**).
    * База Данных: PostgreSQL / MySQL / MongoDB (например, **PostgreSQL**).
    * Инфраструктура: Git, Docker, Облачный хостинг (AWS/Google Cloud/Heroku).
3.2. Требования к производительности:
    * Время ответа API: не более 300 мс для основных запросов.
    * Поддержка не менее 1000 активных пользователей в пиковые часы.
3.3. Требования к безопасности:
    * Использование HTTPS.
    * Хэширование паролей (например, bcrypt).
    * Защита API от XSS и CSRF атак.

4. ТРЕБОВАНИЯ К ДИЗАЙНУ И ИНТЕРФЕЙСУ

4.1. Дизайн: Адаптивный, чистый и минималистичный, ориентированный на удобство использования на мобильных и десктопных устройствах.
4.2. Usability: Интуитивно понятный интерфейс, с минимальным количеством кликов для выполнения основных задач (планирование, добавление продукта).

5. ПРИЕМКА И ТЕСТИРОВАНИЕ

5.1. Тестирование: Модульное, интеграционное и приемочное тестирование.
5.2. Критерии приемки: Соответствие разработанного функционала требованиям данного ТЗ, стабильная работа всех компонентов, отсутствие критических ошибок.

6. ДОПОЛНИТЕЛЬНО

6.1. Масштабируемость: Архитектура должна позволять относительно легкое добавление нового функционала (например, интеграция с фитнес-трекерами).
